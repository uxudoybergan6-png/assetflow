# PERF BASELINE — catalog scale 50 → 500 → 5000

**MUAMMOLAR-1 step 22.** Measured with real numbers, not "fast".
Generated by `scripts/perf-seed-assets.mjs` + `scripts/perf-measure.mjs`.

- **Date:** 2026-07-14
- **Environment:** local API (`apps/api`, Node, 1 process) + local Postgres 16 on the same
  machine (`creative_tools`). Reproduce: `node scripts/perf-seed-assets.mjs <N>` then
  `node scripts/perf-measure.mjs http://localhost:4000 "<N> assets"`.
- **Catalog totals:** N perf-seed rows + 5 pre-existing real templates (so 55 / 505 / 5005).
- **Distribution:** 6 categories (video-templates 30%, graphics 20%, motion-graphics 18%,
  music 14%, sfx 10%, luts 8%), mixed orient (h/v/square), res (45% 4K), Pro (~30%),
  file sizes 0.2–320 MB, realistic names/tags/dimensions.

> ⚠️ **Read the caveats before trusting the absolute milliseconds.** Local Postgres has
> **no network latency**. Production today runs Cloud Run (`europe-west1`) against Neon
> (`us-east-1`) — **~90–110 ms per SQL query on top of every number below**, until the
> Frankfurt DB move (P8 #1, in progress) lands. These numbers are the **post-region-move
> target** and the **relative scaling curve** — both of which are what step 22 is for.

## Results

| Metric (p50) | 50 assets | 500 assets | 5000 assets | Scaling |
|---|---|---|---|---|
| **Catalog list TTFB** | 3.0 ms | 4.7 ms | 4.8 ms | **flat** ✅ |
| Catalog list p95 | 41.7 ms | 42.1 ms | 42.1 ms | flat |
| **Response size (per page, 48 items)** | **38.2 KB** | **38.2 KB** | **38.2 KB** | **flat** ✅ |
| Items per page | 48 | 48 | 48 | paginated |
| **Filter** (`templateType=graphics`) | 1.6 ms | 3.4 ms | 7.9 ms | sub-linear ✅ |
| Filter combo (`type+pro+orient`) | 1.3 ms | 1.8 ms | 3.5 ms | sub-linear |
| **Search** (`q=cinematic`) | 1.9 ms | 4.3 ms | 5.0 ms | sub-linear ⚠️ (see finding 2) |
| **Detail** (`/catalog/:id`) | 0.9 ms · 0.8 KB | 1.2 ms · 0.8 KB | 1.3 ms · 0.8 KB | flat ✅ |
| **Load, 20 concurrent** (wall) | 19 ms · 20/20 ok | 33 ms · 20/20 ok | 25 ms · 20/20 ok | flat ✅ |
| **Load, 50 concurrent** (wall) | 45 ms · 50/50 ok | 46 ms · 50/50 ok | 50 ms · 50/50 ok | flat ✅ |
| **API process RSS** | 178 MB | 208 MB | 220 MB | +42 MB for 100× data ✅ |

**Plugin sync** = the initial catalog load. After step 15 (server-side pagination) both clients
fetch **one page at a time**, so first-paint sync = the "catalog list" row above:
**~38 KB, p50 ~5 ms** (local) regardless of catalog size — the plugin no longer downloads the
whole catalog. (CEP DOM render of the visible window is not measured here — it needs live AE;
step 18 virtualization keeps that to the visible cards only.)

**ETag / edge cache:** the list endpoint returns a weak `ETag` + `Cache-Control:
public, max-age=60, s-maxage=300`. A conditional request (`If-None-Match`) returns **304** →
most catalog opens are edge-cacheable and never reach the DB (step 17).

**What server-side pagination saved** — the OLD "fetch all pages" client (P6 #1) at 5005 assets:
> **51 requests · 5005 items · 3986 KB (~3.9 MB) · 499 ms** in one catalog open.
The new client fetches **1 page · 38 KB** for first paint. That is the difference between a
frozen After Effects and an instant one.

## Findings (stated, not "fixed" — per step 22 rules)

1. **The catalog scales cleanly to 5000.** Response size is flat (38.2 KB/page), TTFB stays
   ~5 ms, 50 concurrent cold opens all succeed. This confirms steps 15 (server pagination),
   16 (slim payload + `assetKeysJson` cache → no per-row storage LIST), 17 (edge cache) and
   4 (filter indexes) are doing their job. **No blocker found at 5000.**

2. **Search is an un-indexed `ILIKE` scan — a scaling watch-item, not yet a problem.**
   `?q=` runs `name/description/catLabel ILIKE '%q%'` + a `tags has` (`buildCatalogWhere`,
   `plugin.ts`). The composite indexes from step 4 do **not** cover `contains` on `name`/
   `description`, so this is a sequential scan. At 5000 it is **5 ms** (fine). It grows
   **linearly** with catalog size (1.9 → 4.3 → 5.0 ms), so beyond ~15–25 k rows it will
   dominate. **Cause:** no trigram / full-text index on the text columns. **Fix when needed
   (not now):** a `pg_trgm` GIN index on `name`/`description`, or route free-text through the
   existing semantic search (`/api/plugin/ai/search`, P5.3). Below ~10 k it is not worth it.

3. **Filter latency grows sub-linearly** (1.6 → 3.4 → 7.9 ms for `graphics`, which is ~20 % of
   rows → ~1000 matched, capped to 48 returned). Index-backed; the growth is from the larger
   matched set, not a table scan. Acceptable; would only warrant attention if it grew linearly.

4. **Memory stays small because the working set is paginated** — RSS +42 MB for a 100× data
   increase. The old full-catalog-in-memory approach (P6 #1) would hold the whole 3.9 MB × per
   connection; pagination removes that.

5. **The dominant production cost is NOT in this measurement: cross-Atlantic DB latency.**
   Every number above is a local-DB best case. Until the Neon Frankfurt move, add ~100 ms per
   query to the list/filter/search rows (the list path is ~1 query, so production ≈ ~105 ms
   TTFB today, collapsing to ~5 ms + edge cache after the move). **The measured architecture is
   ready for 5000; the region move is the remaining launch-blocking latency fix, already
   tracked in P8 #1.**

## Not measurable from here (read from GCP in production)

- **Cloud Run CPU / memory** under real load — read from the Cloud Run metrics dashboard after
  the region move + `max-instances` raise (step 5). The local RSS proxy (178→220 MB) suggests
  the 1 GiB instance has ample headroom for the catalog path at 5000 assets.
- **ffmpeg ingest queue depth** for 5000 clips — that is the bulk-ingest worker's concern
  (step 19 / block H), not the read path measured here.

## How to re-run

```bash
# in one terminal: API against the target DB
node apps/api/dist/index.js
# in another:
node scripts/perf-seed-assets.mjs 500      # 50 | 500 | 5000  (replaces perf-seed rows)
node scripts/perf-measure.mjs http://localhost:4000 "500 assets"
node scripts/perf-seed-assets.mjs --clear  # remove all perf-seed rows
```

Point `API_BASE` at the production/staging URL to measure the real region latency once the
Frankfurt move lands.
